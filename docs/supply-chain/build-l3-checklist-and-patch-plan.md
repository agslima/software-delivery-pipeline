## Goal

Create a credible path from the current `SLSA Build L2` posture to a future point where a `Build L3` claim could be supported by both repository evidence and build-platform evidence.

It is split into:

- controls the repository can patch directly
- controls that require release evidence from actual workflow runs
- controls that depend on platform assurances outside the repository itself

The trusted workflow set for this plan is:

- `.github/workflows/ci-release-gate.yml`
- `.github/workflows/release-build-push-dual-registry.yml`
- `.github/workflows/release-static-risk.yml`
- `.github/workflows/release-dast.yml`
- `.github/workflows/gitops-enforce.yml`

## Build L3 Checklist

Status legend:

- `Done`: implemented and evidenced in the repository
- `Partial`: repository work exists, but evidence or platform guarantees are incomplete
- `Open`: not yet addressed enough to support a Build L3 claim

### A. Prevent Runs From Influencing One Another

| Checklist item | Current status | Current evidence | What still blocks a Build L3 claim |
| :--- | :--- | :--- | :--- |
| Trusted release and promotion workflows run on ephemeral hosted runners | `Partial` | Trusted workflows use GitHub-hosted runners and avoid long-lived self-hosted runners | The repo cannot prove platform-level run isolation on `ubuntu-latest`; this remains a hosted-platform assumption. |
| Trusted workflows use pinned third-party actions | `Done` | `scripts/check-workflow-input-provenance.py` passes on trusted workflows | Keep enforcement and regression tests in place. |
| Trusted workflows use digest-pinned deployable images | `Done` | Digest artifacts and GitOps promotion use immutable digests | Continue to treat digest-only promotion as non-negotiable. |
| Trusted workflows avoid shared mutable cache state | `Partial` | Release builds use `no-cache: true`; reproducibility pilot uses `--no-cache` | The repo does not prove that the underlying hosted builder and service-side caches cannot create cross-run influence. |
| Release build inputs are normalized enough for reproducibility checks | `Partial` | Backend reproducibility pilot exists in `ci-release-gate.yml`; the 2026-05-08 backend pilot record shows `status: mismatch` | Only one image has a non-blocking pilot; no successful pilot evidence is recorded yet; the recorded mismatch requires investigation before the result can support stronger SLSA evidence. |
| Trusted workflows minimize live dependency fetches | `Partial` | Base images are digest-pinned; some installers are pinned and verified | `npm ci`, Trivy DB access, and registry interactions still rely on live external services. |

### B. Prevent Provenance-Signing Material From Being Accessible To User-Defined Build Steps

| Checklist item | Current status | Current evidence | What still blocks a Build L3 claim |
| :--- | :--- | :--- | :--- |
| Provenance is generated by a platform-controlled mechanism | `Partial` | SLSA provenance is generated by `slsa-github-generator` reusable workflow calls in `ci-release-gate.yml` | The reusable workflow ref is still version-tag-based rather than commit-SHA-based. |
| Signing and provenance happen after gated build/scanning stages | `Done` | Build, Trivy, and DAST stages complete before downstream verification and promotion | Preserve this sequencing. |
| GitHub token permissions are scoped per trusted job | `Done` | Trusted workflows declare job-specific `permissions:` blocks | Continue periodic review of scopes. |
| User-defined build shell steps do not directly hold long-lived signing keys | `Done` | Keyless OIDC/cosign flow is used; no committed long-lived signing key material is present in workflows | Preserve keyless pattern. |
| Promotion verifies builder identity and provenance before trust is granted | `Done` | `gitops-enforce.yml` verifies signature, attestations, builder ID, source URI, and source tag | Preserve fail-closed verification. |
| Repository can prove provenance-signing secret material is inaccessible to user-defined steps under platform failure modes | `Open` | Repository docs explicitly do not claim this | This requires stronger platform-level guarantees or external audit evidence, not just workflow edits. |

## Patch Plan

### Phase 0: Keep Current Guardrails Green

Objective:
ensure the existing repo-side controls stay enforced and auditable.

Checklist:

- [x] Keep `scripts/check-workflow-input-provenance.py` passing on trusted workflows.
- [x] Keep `step-security/harden-runner` present in release, scan, reproducibility-pilot, and GitOps promotion jobs.
- [x] Keep GitOps promotion fail-closed on provenance and attestation verification.
- [x] Keep digest-only promotion and deployment behavior intact.

Expected repo touchpoints:

- `.github/workflows/ci-pr-validation.yml`
- `.github/workflows/ci-release-gate.yml`
- `.github/workflows/release-build-push-dual-registry.yml`
- `.github/workflows/release-static-risk.yml`
- `.github/workflows/release-dast.yml`
- `.github/workflows/gitops-enforce.yml`

### Phase 1: Eliminate Remaining Easy Mutable Inputs

Objective:
reduce repository-level mutable inputs that are still easy to patch without platform changes.

Checklist:

- [x] Evaluate whether `slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.1.0` can be pinned by commit SHA instead of release tag in `ci-release-gate.yml`: currently, the SLSA reusable workflow does not support being pinned by commit SHA; the tag-based ref is the required approach for slsa-github-generator.
- [x] Continue converting installer steps to checksum-verified or mirrored sources where possible.
- [x] Review whether any trusted workflow still implicitly depends on mutable defaults such as `latest` resolution or unpinned runtime images.

Primary patch targets:

- `.github/workflows/ci-release-gate.yml`
- `.github/workflows/gitops-enforce.yml`
- `docs/trusted-workflow-input-inventory.md`

### Phase 2: Turn Reproducibility Pilots Into Evidence

Objective:
move from pilot definition to actual reproducibility evidence from release runs.

Checklist:

- [x] Collect successful `reproducibility-check` evidence from a real `Release` run (at minimum for backend).
- [x] Record the first pilot outcome in durable governance evidence.
- [x] Decide whether the backend pilot should stay advisory or become a stronger gate.
- [x] Extend the same pilot pattern to frontend or worker only if backend results are stable.

Primary patch and evidence targets:

- `.github/workflows/ci-release-gate.yml`
- `docs/supply-chain/reproducibility-pilot-backend.md`
- release artifacts `reproducibility-pilot-backend/report.json` and `summary.md`
- pilot evidence record in `docs/supply-chain/reproducibility-pilot-backend.md`

### Phase 3: Reduce Cross-Run Influence From External Inputs

Objective:
limit the ability of live external systems to introduce nondeterminism or shared influence.

Checklist:

- [ ] Design a mirrored or pre-fetched package-source approach for `npm ci` in `release-build-push-dual-registry.yml`.
- [ ] Decide whether Trivy DB mirroring is warranted by operational history.
- [ ] Review whether release builds should avoid or further constrain remote pulls beyond current digest pinning.
- [x] Investigate the 2026-05-08 backend pilot mismatch before expanding reproducibility checks to additional images or using the backend result as stronger SLSA evidence.

Primary patch targets:

- `.github/workflows/release-build-push-dual-registry.yml`
- `.github/workflows/release-static-risk.yml`
- `docs/trusted-workflow-input-inventory.md`

### Phase 4: Resolve Platform-Level Builder Isolation Questions

Objective:
close the gaps that cannot be solved by repository code alone.

Checklist:

- [ ] Decide whether `ubuntu-latest` with current controls is sufficient for the project’s assurance target, or whether a stronger builder model is required.
- [ ] Obtain platform-level evidence or an internal decision record addressing run-to-run isolation expectations.
- [ ] Obtain platform-level evidence or an internal decision record addressing whether provenance-signing material is inaccessible to user-defined build steps.
- [ ] Record the decision in a new ADR or governance addendum before making any Build L3 claim.

Expected evidence rather than code:

- ADR or governance decision record
- platform assessment notes
- release evidence showing the repo-side controls stayed effective during the assessment period

## Claim Gate

Use this gate before changing the repo’s posture statement to Build L3:

- [ ] Repository-side guardrails remain green.
- [ ] Reproducibility pilot has real successful evidence from release runs.
- [ ] Mutable-input exceptions are reduced or explicitly accepted with rationale.
- [ ] A platform-level decision or evidence package exists for run isolation.
- [ ] A platform-level decision or evidence package exists for provenance-signing material isolation.
